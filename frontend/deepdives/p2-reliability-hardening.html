P2

Answer consistency and accuracy

Reliability here means users get the same grounded answer for the same question, with correct citations and fewer contradictory outputs.

Current vs Recommended

What they are doing now

Quality is checked informally, so similar questions can return different wording, missing citations, or mixed values.

More reliable way

Add answer-quality gates: fixed evaluation set, citation validation, and conflict-aware answer rules before publishing refresh changes.

Why this works

This shifts reliability from system uptime to user trust: repeatable and evidence-backed answers reduce rework and escalations.

Quality control pattern

Golden question set

- 30-50 representative questions
- include numeric/policy questions
- expected answer + expected citations
- compare results after each change

Citation gate

- numeric claims must cite source chunks
- reject answers with missing evidence
- flag if answer cites irrelevant source

Conflict handling

- if multiple values found, separate them
- classify value type (draft/final/etc.)
- avoid forcing one value silently

Where the golden question set should live

Recommended location

Keep it in source control beside the ingestion or evaluation assets so it changes with the solution, not in a personal spreadsheet.

Practical formats

Use JSON for structured automation or CSV for easier editing by content owners. Either works as long as the schema stays stable.

Ownership

Product/content owners define the questions. Platform or engineering automates execution and score reporting.

Example schema

JSON example

{
  "id": "reg-001",
  "question": "What is the approved late fee percentage?",
  "mustCite": true,
  "expectedSources": ["late-fee-policy.pdf"],
  "expectedAnswerContains": ["approved", "late fee"],
  "notes": "Numeric/policy claim"
}

Minimum fields

- id
- question
- mustCite
- expectedSources
- expectedAnswerContains
- notes

Optional scoring fields

- forbiddenSources
- allowedValuePatterns
- topic
- owner
- severity

How to implement

  1. Start with 20-30 real business questions that already have known answers in the source material.
  2. Store them in a versioned JSON or CSV file in the repo or evaluation project.
  3. After every indexing, chunking, or model change, run the same set through Copilot Studio test pane, an agent playground, or an automated API harness.
  4. Score each result for citation presence, citation relevance, answer consistency, and contradiction handling.
  5. Promote changes only if the score stays within the agreed threshold.

Tip: start with your top 20 business-critical questions and optimize for repeatability first. Add new questions whenever users challenge an answer in production.

How to run it in practice

  1. Run the golden questions before making a change and save the baseline results.
  2. Apply the change, such as lower overlap or a different embedding model.
  3. Run the same questions again.
  4. Compare output side by side: did the answer stay correct, did citations stay relevant, did contradictions appear?
  5. Record pass/fail and keep the result set for audit and troubleshooting.

A simple first version can be manual in Copilot Studio. A better long-term version is an automated harness that posts each question to the agent and scores the response.
