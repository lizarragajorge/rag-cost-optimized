    Cost comparison: small vs large embeddings

    Same corpus, same churn, same refresh cadence. Model choice alone can materially change annual cost.

    Why start with 1,200,000,000 tokens?

    This default is meant to approximate a mid-sized enterprise knowledge base, not a small demo corpus. A practical rule of thumb is that 1 GB of English-heavy text often lands around 200M to 250M tokens after extraction, depending on formatting noise, tables, and duplication.

    • 100 GB raw files does not mean 100 GB of useful text. Most teams size from extracted text, not storage size.
    • 5 GB extracted text at roughly 240M tokens per GB is about 1.2B tokens.
    • 10 GB extracted text is often closer to 2.1B to 2.4B tokens.
    • Heavy overlap and OCR noise can push the effective token count higher than expected.

    If you only know document storage size, start by estimating the extracted-text portion first, then convert to tokens. This keeps the calculator closer to real embedding cost.

    Recommended change plan

    1. Cost controls first

    1. Use smaller embedding model unless eval proves otherwise.
    2. Reduce overlap in chunking to avoid duplicate tokens.
    3. Strip low-value OCR/noise before embedding.

    2. Answer reliability guardrails

    1. Use a fixed evaluation set to test consistency after every indexing change.
    2. Require citation-backed answers for numeric or policy questions.
    3. Track accuracy and contradiction rate in weekly review.

    3. Team operating model

    1. Copilot Studio stays focused on authoring and responses.
    2. Azure AI Search handles indexing and grounding configuration close to the knowledge source.
    3. Teams validate answer quality in Copilot Studio using the same golden question set after changes.

    • Improves both cost and answer precision

    Best practices: low-signal extracted text

    Clean low-value extraction artifacts before chunking to lower token volume without reducing useful retrieval context.

    Common low-signal patterns

    • Repeated image and attachment marker lines
    • Boilerplate headers and footers repeated on every page
    • OCR artifacts, scanning noise, and broken wraps
    • Navigation fragments and duplicate whitespace blocks

    Recommended implementation pattern

    extract -> cleanText -> chunk -> embed -> index

cleanText rules:
- remove repeated marker lines
- normalize whitespace and wraps
- preserve semantic headings
- keep labels needed for citations

    Apply cleanup before chunking and hashing so incremental runs can reuse existing embeddings reliably.

    • Compare token volume before and after cleanup as a release check.
    • Validate that citations still point to expected sources for key questions.
    • Tune rules in small steps and re-run the same golden questions each time.

    Best practices: answer-quality gates

    Use fixed gates to decide whether indexing, chunking, or model changes are safe to promote.

    Gate categories

    • Answer correctness
    • Citation presence
    • Citation relevance
    • Contradiction handling
    • Consistency across repeated prompts

    Suggested thresholds

    qualityGate = {
  citationPassRate: ">= 95%",
  contradictionRate: "<= 2%",
  criticalQuestionPassRate: ">= 95%"
}

    Store golden questions in source control (JSON or CSV) and run the same set after every material change.

    1. Run golden questions in Copilot Studio test pane after each change.
    2. Score pass/fail for correctness, citations, relevance, and contradiction handling.
    3. Block promotion if thresholds are missed and add failures back into the question set.
